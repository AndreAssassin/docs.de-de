# <a name="writing-secure-dynamic-sql-in-sql-server"></a><span data-ttu-id="f1041-102">Schreiben von sicherem dynamischen SQL in SQL Server</span><span class="sxs-lookup"><span data-stu-id="f1041-102">Writing Secure Dynamic SQL in SQL Server</span></span>
<span data-ttu-id="f1041-103">Die Einschleusung von SQL-Befehlen (SQL Injection-Angriff) ist eine Möglichkeit, Anwendungen anzugreifen. Dabei werden anstelle einer gültigen Eingabe Transact-SQL-Anweisungen in den Code eingeschleust.</span><span class="sxs-lookup"><span data-stu-id="f1041-103">SQL Injection is the process by which a malicious user enters Transact-SQL statements instead of valid input.</span></span> <span data-ttu-id="f1041-104">Wenn die Eingabe direkt und ohne Validierung an den Server weitergeleitet wird und die Anwendung den eingeschleusten Code ausführt, können Daten beschädigt oder sogar zerstört werden.</span><span class="sxs-lookup"><span data-stu-id="f1041-104">If the input is passed directly to the server without being validated and if the application inadvertently executes the injected code, the attack has the potential to damage or destroy data.</span></span>  
  
 <span data-ttu-id="f1041-105">Alle Prozeduren, die SQL-Anweisungen konstruieren, sollten auf Schwachstellen überprüft werden, die die Einschleusung von SQL-Befehlen ermöglichen, da SQL Server alle syntaktisch gültigen Abfragen ungeprüft ausführt.</span><span class="sxs-lookup"><span data-stu-id="f1041-105">Any procedure that constructs SQL statements should be reviewed for injection vulnerabilities because SQL Server will execute all syntactically valid queries that it receives.</span></span> <span data-ttu-id="f1041-106">Selbst parametrisierte Daten können von einem geschickten und entschlossenen Angreifer manipuliert werden.</span><span class="sxs-lookup"><span data-stu-id="f1041-106">Even parameterized data can be manipulated by a skilled and determined attacker.</span></span> <span data-ttu-id="f1041-107">Bei Verwendung von dynamischem SQL sollten Sie Ihre Befehle unbedingt parametrisieren und auf keinen Fall Parameterwerte direkt in die Abfragezeichenfolge aufnehmen.</span><span class="sxs-lookup"><span data-stu-id="f1041-107">If you use dynamic SQL, be sure to parameterize your commands, and never include parameter values directly into the query string.</span></span>  
  
## <a name="anatomy-of-a-sql-injection-attack"></a><span data-ttu-id="f1041-108">Anatomie eines SQL Injection-Angriffs</span><span class="sxs-lookup"><span data-stu-id="f1041-108">Anatomy of a SQL Injection Attack</span></span>  
 <span data-ttu-id="f1041-109">Beim Einschleusen wird eine Textzeichenfolge vorzeitig beendet, und es wird ein neuer Befehl angefügt.</span><span class="sxs-lookup"><span data-stu-id="f1041-109">The injection process works by prematurely terminating a text string and appending a new command.</span></span> <span data-ttu-id="f1041-110">Da der eingeschleuste Befehl zusätzliche Zeichenfolgen enthalten kann, die vor dessen Ausführung angefügt wurden, beendet der Angreifer die eingeschleuste Zeichenfolge mit einer Kommentarmarkierung ("--").</span><span class="sxs-lookup"><span data-stu-id="f1041-110">Because the inserted command may have additional strings appended to it before it is executed, the malefactor terminates the injected string with a comment mark "--".</span></span> <span data-ttu-id="f1041-111">Der Text, der dieser Kommentarmarkierung folgt, wird bei der Ausführung ignoriert.</span><span class="sxs-lookup"><span data-stu-id="f1041-111">Subsequent text is ignored at execution time.</span></span> <span data-ttu-id="f1041-112">Durch Eingabe eines Semikolons (;) als Trennzeichen können auch mehrere Befehle eingeschleust werden.</span><span class="sxs-lookup"><span data-stu-id="f1041-112">Multiple commands can be inserted using a semicolon (;) delimiter.</span></span>  
  
 <span data-ttu-id="f1041-113">Sofern der SQL-Code syntaktisch korrekt ist, kann die Manipulation nicht programmgesteuert erkannt werden.</span><span class="sxs-lookup"><span data-stu-id="f1041-113">As long as injected SQL code is syntactically correct, tampering cannot be detected programmatically.</span></span> <span data-ttu-id="f1041-114">Sie müssen daher alle Benutzereingaben validieren und Code sorgfältig prüfen, der konstruierte SQL-Befehle im verwendeten Server ausführt.</span><span class="sxs-lookup"><span data-stu-id="f1041-114">Therefore, you must validate all user input and carefully review code that executes constructed SQL commands in the server that you are using.</span></span> <span data-ttu-id="f1041-115">Verketten Sie nie Benutzereingaben, die nicht validiert wurden.</span><span class="sxs-lookup"><span data-stu-id="f1041-115">Never concatenate user input that is not validated.</span></span> <span data-ttu-id="f1041-116">Die Zeichenfolgenverkettung ist der primäre Ansatzpunkt für die Einschleusung von Skriptcode.</span><span class="sxs-lookup"><span data-stu-id="f1041-116">String concatenation is the primary point of entry for script injection.</span></span>  
  
 <span data-ttu-id="f1041-117">Beachten Sie Folgendes:</span><span class="sxs-lookup"><span data-stu-id="f1041-117">Here are some helpful guidelines:</span></span>  
  
-   <span data-ttu-id="f1041-118">Erstellen Sie keine Transact-SQL-Anweisungen direkt aus Benutzereingaben. Validieren Sie Benutzereingaben mit gespeicherten Prozeduren.</span><span class="sxs-lookup"><span data-stu-id="f1041-118">Never build Transact-SQL statements directly from user input; use stored procedures to validate user input.</span></span>  
  
-   <span data-ttu-id="f1041-119">Validieren Sie Benutzereingaben durch Prüfen des Typs, der Länge, des Formats und des Bereichs.</span><span class="sxs-lookup"><span data-stu-id="f1041-119">Validate user input by testing type, length, format, and range.</span></span> <span data-ttu-id="f1041-120">Verwenden Sie die Transact-SQL-QUOTENAME()-Funktion, um Systemnamen zu maskieren, oder die REPLACE()-Funktion, um Zeichen in einer Zeichenfolge zu maskieren.</span><span class="sxs-lookup"><span data-stu-id="f1041-120">Use the Transact-SQL QUOTENAME() function to escape system names or the REPLACE() function to escape any character in a string.</span></span>  
  
-   <span data-ttu-id="f1041-121">Implementieren Sie für jede Anwendungsebene mehrere Validierungsschichten.</span><span class="sxs-lookup"><span data-stu-id="f1041-121">Implement multiple layers of validation in each tier of your application.</span></span>  
  
-   <span data-ttu-id="f1041-122">Prüfen Sie die Größe und den Datentyp der Eingabe, und sorgen Sie für die Einhaltung der festgelegten Grenzwerte.</span><span class="sxs-lookup"><span data-stu-id="f1041-122">Test the size and data type of input and enforce appropriate limits.</span></span> <span data-ttu-id="f1041-123">Dies kann helfen, vorsätzliche Pufferüberläufe zu verhindern.</span><span class="sxs-lookup"><span data-stu-id="f1041-123">This can help prevent deliberate buffer overruns.</span></span>  
  
-   <span data-ttu-id="f1041-124">Prüfen Sie den Inhalt von Zeichenfolgenvariablen, und akzeptieren Sie nur erwartete Werte.</span><span class="sxs-lookup"><span data-stu-id="f1041-124">Test the content of string variables and accept only expected values.</span></span> <span data-ttu-id="f1041-125">Lehnen Sie Einträge ab, die Binärdaten, Maskierungssequenzen und Kommentarzeichen enthalten.</span><span class="sxs-lookup"><span data-stu-id="f1041-125">Reject entries that contain binary data, escape sequences, and comment characters.</span></span>  
  
-   <span data-ttu-id="f1041-126">Validieren Sie beim Arbeiten mit XML-Dokumenten alle eingegebenen Daten anhand ihres Schemas.</span><span class="sxs-lookup"><span data-stu-id="f1041-126">When you are working with XML documents, validate all data against its schema as it is entered.</span></span>  
  
-   <span data-ttu-id="f1041-127">Sorgen Sie dafür, dass in Umgebungen mit mehreren Ebenen alle Daten vor dem Eintritt in die vertrauenswürdige Zone validiert werden.</span><span class="sxs-lookup"><span data-stu-id="f1041-127">In multi-tiered environments, all data should be validated before admission to the trusted zone.</span></span>  
  
-   <span data-ttu-id="f1041-128">Akzeptieren Sie keine der folgenden Zeichenfolgen in Feldern, die von denen Dateinamen erstellt werden können: AUX, CLOCK$, COM1 bis COM8, CON, CONFIG$, LPT1 bis LPT8, NUL und PRN.</span><span class="sxs-lookup"><span data-stu-id="f1041-128">Do not accept the following strings in fields from which file names can be constructed: AUX, CLOCK$, COM1 through COM8, CON, CONFIG$, LPT1 through LPT8, NUL, and PRN.</span></span>  
  
-   <span data-ttu-id="f1041-129">Verwenden Sie für die Typprüfung und die Längenvalidierung <xref:System.Data.SqlClient.SqlParameter>-Objekte mit gespeicherten Prozeduren und Befehlen.</span><span class="sxs-lookup"><span data-stu-id="f1041-129">Use <xref:System.Data.SqlClient.SqlParameter> objects with stored procedures and commands to provide type checking and length validation.</span></span>  
  
-   <span data-ttu-id="f1041-130">Verwenden Sie im Clientcode <xref:System.Text.RegularExpressions.Regex>-Ausdrücke, um ungültige Zeichen herauszufiltern.</span><span class="sxs-lookup"><span data-stu-id="f1041-130">Use <xref:System.Text.RegularExpressions.Regex> expressions in client code to filter invalid characters.</span></span>  
  
## <a name="dynamic-sql-strategies"></a><span data-ttu-id="f1041-131">Strategien bei der Verwendung von dynamischem SQL</span><span class="sxs-lookup"><span data-stu-id="f1041-131">Dynamic SQL Strategies</span></span>  
 <span data-ttu-id="f1041-132">Beim Ausführen dynamisch erstellter SQL-Anweisungen in Ihrem prozeduralen Code wird die Besitzkette unterbrochen, sodass SQL Server die Berechtigungen des Aufrufers anhand der Objekte prüfen muss, auf die die dynamischen SQL-Anweisungen zugreifen.</span><span class="sxs-lookup"><span data-stu-id="f1041-132">Executing dynamically created SQL statements in your procedural code breaks the ownership chain, causing SQL Server to check the permissions of the caller against the objects being accessed by the dynamic SQL.</span></span>  
  
 <span data-ttu-id="f1041-133">SQL Server verfügt über Methoden, um Benutzern mithilfe von gespeicherten Prozeduren und benutzerdefinierten Funktionen, die dynamischen SQL-Code ausführen, den Zugriff auf Daten zu gewähren.</span><span class="sxs-lookup"><span data-stu-id="f1041-133">SQL Server has methods for granting users access to data using stored procedures and user-defined functions that execute dynamic SQL.</span></span>

### <a name="execute-as"></a><span data-ttu-id="f1041-136">EXECUTE AS</span><span class="sxs-lookup"><span data-stu-id="f1041-136">EXECUTE AS</span></span>  
 <span data-ttu-id="f1041-137">Die EXECUTE AS-Klausel ersetzt die Berechtigungen des Aufrufers durch die Berechtigungen des in der EXECUTE AS-Klausel angegebenen Benutzers.</span><span class="sxs-lookup"><span data-stu-id="f1041-137">The EXECUTE AS clause replaces the permissions of the caller with that of the user specified in the EXECUTE AS clause.</span></span> <span data-ttu-id="f1041-138">Geschachtelte gespeicherte Prozeduren oder Trigger werden im Sicherheitskontext des Proxybenutzers ausgeführt.</span><span class="sxs-lookup"><span data-stu-id="f1041-138">Nested stored procedures or triggers execute under the security context of the proxy user.</span></span> <span data-ttu-id="f1041-139">Bei Anwendungen, die sich auf zeilenbasierte Sicherheit stützen oder eine Überwachung verlangen, kann dies zum Abbruch führen.</span><span class="sxs-lookup"><span data-stu-id="f1041-139">This can break applications that rely on row-level security or require auditing.</span></span> <span data-ttu-id="f1041-140">Einige Funktionen, die die Identität des Benutzers zurückgeben, geben den in der EXECUTE AS-Klausel angegebenen Benutzer und nicht den ursprünglichen Aufrufer zurück.</span><span class="sxs-lookup"><span data-stu-id="f1041-140">Some functions that return the identity of the user return the user specified in the EXECUTE AS clause, not the original caller.</span></span> <span data-ttu-id="f1041-141">Der Ausführungskontext geht erst dann wieder auf den ursprünglichen Aufrufer über, wenn die Ausführung der Prozedur abgeschlossen wurde oder wenn eine REVERT-Anweisung ausgegeben wird.</span><span class="sxs-lookup"><span data-stu-id="f1041-141">Execution context is reverted to the original caller only after execution of the procedure or when a REVERT statement is issued.</span></span>  
  
### <a name="certificate-signing"></a><span data-ttu-id="f1041-142">Signieren mit Zertifikaten</span><span class="sxs-lookup"><span data-stu-id="f1041-142">Certificate Signing</span></span>  
 <span data-ttu-id="f1041-143">Bei der Ausführung einer mit einem Zertifikat signierten gespeicherten Prozedur werden die dem Zertifikatsbenutzer gewährten Berechtigungen mit den Berechtigungen des Aufrufers zusammengeführt.</span><span class="sxs-lookup"><span data-stu-id="f1041-143">When a stored procedure that has been signed with a certificate executes, the permissions granted to the certificate user are merged with those of the caller.</span></span> <span data-ttu-id="f1041-144">Der Ausführungskontext bleibt identisch. Der Zertifikatsbenutzer nimmt nicht die Identität des Aufrufers an.</span><span class="sxs-lookup"><span data-stu-id="f1041-144">The execution context remains the same; the certificate user does not impersonate the caller.</span></span> <span data-ttu-id="f1041-145">Für das Signieren gespeicherter Prozeduren müssen verschiedene Schritte ausgeführt werden.</span><span class="sxs-lookup"><span data-stu-id="f1041-145">Signing stored procedures requires several steps to implement.</span></span> <span data-ttu-id="f1041-146">Jedes Mal, wenn die Prozedur geändert wird, muss sie neu signiert werden.</span><span class="sxs-lookup"><span data-stu-id="f1041-146">Each time the procedure is modified, it must be re-signed.</span></span>
